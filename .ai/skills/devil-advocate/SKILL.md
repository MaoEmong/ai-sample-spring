# Devil's Advocate

생성된 코드나 설계를 의도적으로 의심하고 비판적으로 검토하여 버그,
설계 문제, 성능 문제, 보안 취약점, 엣지 케이스를 찾는 리뷰 스킬이다.

## 언제 사용하는가

- 사용자가 `/devil-advocate`를 입력했을 때
- 코드 리뷰, 설계 검토, 품질 점검을 요청했을 때
- AI가 코드를 생성한 직후, 구현 결과를 비판적으로 검토해야 할 때
- AI가 직접 코드를 작성하거나 수정하는 작업에서 구현 직후 자기 검토가 필요할 때
- "코드 리뷰해줘", "문제 없는지 봐줘", "이 설계 괜찮아?", "리스크가 뭐야?" 같은 요청이 들어왔을 때

## 왜 이 스킬이 필요한가

AI는 자신이 만든 결과물에 확신 편향(confirmation bias)을 가지기 쉽다.
별도의 비판 단계가 없으면 다음 문제가 발생할 수 있다.

- 숨은 버그
- 엣지 케이스 미처리
- 비효율적인 알고리즘
- 보안 취약점
- 유지보수 어려움

이 스킬은 일부러 반대 입장에서 문제점을 찾아 코드 품질을 높인다.

## 리뷰 원칙

### 항상 의심하기

코드가 정상 동작한다고 가정하지 않는다.

다음 질문으로 접근한다.

```text
이 코드가 실패한다면 왜 실패할까?
```

### 최악의 상황 가정

정상 입력이 아니라 **비정상 입력**을 먼저 생각한다.

예:

```text
null
빈 데이터
대량 데이터
동시 요청
네트워크 실패
```

### 공격적이되 건설적으로

문제를 찾는 것이 목적이지만 비난이 아니라 **개선 방향**을 함께 제시한다.

## 리뷰 흐름

아래 단계는 기본 흐름이다.

### Phase 1: 기능 정확성

코드가 요구사항을 **정확히 구현했는지** 검증한다.

질문 기법:

- 요구사항과 다른 동작을 할 가능성은 없는가?
- 숨은 버그가 존재할 가능성은?
- 예외 처리가 빠진 부분은 없는가?

### Phase 2: 엣지 케이스

일반적인 입력이 아니라 **극단적인 상황**을 검토한다.

질문 기법:

- 입력값이 null이면 어떻게 되는가?
- 데이터가 0개일 때 동작하는가?
- 데이터가 매우 많으면 성능 문제는 없는가?

### Phase 3: 성능

시간 복잡도와 확장성을 검토한다.

질문 기법:

- 시간 복잡도는 적절한가?
- 대량 트래픽에서도 동작하는가?
- 불필요한 반복 연산은 없는가?

### Phase 4: 설계 품질

코드 구조와 유지보수성을 검토한다.

질문 기법:

- 이 코드는 읽기 쉬운가?
- 책임이 명확하게 분리되어 있는가?
- 더 단순한 구조로 만들 수 있는가?

### Phase 5: 보안

보안 취약점을 탐색한다.

질문 기법:

- 입력 검증이 충분한가?
- SQL Injection 가능성은 없는가?
- 민감한 데이터가 노출될 가능성은 없는가?

## 리뷰 시작 방법

사용자가 `/devil-advocate`를 입력하면 다음과 같이 시작한다.

```text
좋습니다. Devil's Advocate 리뷰를 시작하겠습니다.

이 코드를 의심하는 관점에서 검토하겠습니다.
버그 가능성, 엣지 케이스, 성능 문제, 설계 품질을 중심으로 살펴보겠습니다.
```

## 리뷰 결과 형식

리뷰 결과는 반드시 다음 형식을 사용한다.

```markdown
## Devil's Advocate Review

### 발견된 문제

1. [문제 제목]
[문제 설명]

원인
[왜 문제가 발생하는지]

개선 제안
[수정 방법]

---

2. [문제 제목]
...

### 추가 개선 제안

- 코드 가독성 개선
- 테스트 케이스 추가
- 리팩토링 제안
```

## 리뷰 종료

리뷰가 끝나면 다음 질문으로 마무리한다.

```text
위 문제들을 수정한 개선 버전을 생성할까요?
```

## 주의사항

- 리뷰 단계에서는 새로운 기능을 추가하지 않는다.
- AI가 코드를 작성하거나 수정한 작업이라면 사용자 요청이 없어도 구현 직후 이 스킬을 수행한다.
- 발견된 문제는 반드시 **원인과 해결 방법을 함께 제시한다.**
- 문제 수에 집착하지 않는다. 중요한 문제 몇 개만 찾아도 충분하다.
